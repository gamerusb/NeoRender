# NeoRender Pro: ICP, конкуренция и SaaS-зрелость

Документ реализует план сравнения с рынком и сопоставления с кодовой базой. Файл плана в `.cursor/plans` не дублируется.

---

## 1. ICP и регион

**Идеальный клиент (ICP), выведенный из продукта**

- **Кто:** команды и индивидуальные продюсеры, которым нужен **сквозной цикл**: подготовка уникализированного видео (FFmpeg-пресеты, шаблоны монтажа, субтитры), опционально **загрузка на YouTube** и сопутствующая **автоматизация браузера** (антидетект-профили, warmup), плюс **аналитика** по опубликованным роликам.
- **За что платят:** экономия времени на повторяющейся обработке и постинге, масштабирование объёма роликов без линейного роста ручной работы.
- **Технический профиль:** готовность держать **локальный сервер** (FastAPI + очередь задач) и окружение с **FFmpeg/GPU** и при необходимости **AdsPower** и т.п. — это ближе к **desktop / private agent**, чем к чистому browser-only SaaS.

**Регион**

- Интерфейс и сообщения в коде преимущественно на **русском**; продукт при этом **не привязан** к одной юрисдикции технически. Для go-to-market регион задаётся вами (СНГ vs глобальный англоязычный Shorts-рынок). В документе зафиксирован **двухконтурный** взгляд: конкуренты на **глобальном** SaaS-рынке видео-автоматизации и **нишевые** десктоп/агент решения (часто без публичного маркетинга).

## 2. Короткий список конкурентов и SaaS-чеклист

Подобраны **5** представителей по двум осям из плана: **API/облачный рендер** и **AI / long-to-short**. Прямые аналоги с тем же стеком (локальный AdsPower + ваш пайплайн) в открытом доступе встречаются реже — их нужно дополнять по вашему сегменту вручную.

| Критерий | Shotstack | Creatomate | Opus Clip | Klap | Renderly |
|----------|-------------|------------|-----------|------|----------|
| **Позиционирование** | Программируемый видео-API, батчи | Шаблоны + API, персонализация | Long → Shorts, AI-нарезка | Long → Shorts, соцсети | Рендер по API, scale |
| **Облачный рендер без агента** | Да | Да | Да | Да | Да |
| **Типичная монетизация** | Подписка / usage | Подписка / credits | Подписка / минуты | Подписка | Credits / usage |
| **Публичный API + документация** | Сильная сторона | Сильная сторона | Ограниченно vs «чистым» API-вендорам | Ограниченно | API-first |
| **Webhooks / асинхронные джобы** | Ожидаемы у API-вендоров | Ожидаемы | Часто облачный пайплайн без вашего воркера | Аналогично | Акцент на массовый рендер |
| **Локальный FFmpeg + антидетект у клиента** | Нет | Нет | Нет | Нет | Нет |
| **Чем часто «сильнее» для классического SaaS-покупателя** | Масштаб, отсутствие установки, предсказуемый биллинг | Гибкие шаблоны, интеграции | Онбординг, магия AI-нарезки | UX для создателей | Простой scale рендера |

**Вывод по конкурентам:** на типичных критериях **облачного SaaS** (без установки, metering, статус-платформа, интеграции) перечисленные игроки **часто выглядят сильнее**, потому что их продукт изначально **hosted**. NeoRender конкурирует не тем же самым unit-экономическим предложением, а **контролем данных, локальным рендером и связкой с антидетект/YouTube** — это нужно явно продавать и дожимать в продукте (прозрачность, доверие, приватность).

## 3. Сопоставление чеклиста с кодовой базой

Ниже — фактическое состояние репозитория NeoRender Pro (без правок кода в рамках этой задачи).

| Слой SaaS | У конкурентов (типично) | В NeoRender сейчас |
|-----------|-------------------------|---------------------|
| **Мультитенантность** | Org, изоляция данных | Есть: `X-Tenant-ID`, JWT с `tenant_id`, отдельный `AutomationPipeline` на tenant ([`api_server.py`](../api_server.py), [`core/tenancy.py`](../core/tenancy.py)). |
| **Аутентификация** | Email/OAuth, SSO для Enterprise | Есть: JWT, bcrypt, роли, регистрация/логин ([`core/auth.py`](../core/auth.py)). |
| **Планы и лимиты** | Связь оплаты ↔ entitlements | **Частично:** словарь `_PLAN_LIMITS` и поле `plan` у пользователя ([`core/database.py`](../core/database.py)). Страница биллинга в UI описывает тарифы ([`frontend/src/pages/cabinet/BillingPage.tsx`](../frontend/src/pages/cabinet/BillingPage.tsx)). **Usage в API сейчас заглушка:** в `_user_public` всегда `tasks_today: 0`, `profiles_used: 0`, и т.д. ([`core/database.py`](../core/database.py) около `2594–2613`). Реального учёта использования и блокировки по плану на уровне создания задач в одном `grep` по `api_server` не видно. |
| **Оплата (Stripe, Paddle, …)** | Обязательный контур | **Не найдено** в Python/TS поиску по ключевым словам. |
| **Webhooks завершения задач** | Стандарт для API-платформ | **Не найдено** как продуктовая фича. |
| **Rate limiting** | Per API key / per tenant | Есть **in-memory** лимиты по tenant для отдельных scope ([`api_server.py`](../api_server.py), `_is_rate_limited`). При перезапуске процесса счётчики сбрасываются. |
| **SSO / SAML / OIDC** | Enterprise | **Не найдено**. |
| **Публичный OpenAPI как артефакт** | Часто да (FastAPI может отдавать схему) | FastAPI используется; отдельная публикация схемы в документе не фиксировалась. |
| **Status page / SLA** | Маркетинг + операционка | Не в репозитории как часть приложения. |

**Итог gap analysis:** задел **tenant + JWT + роли + модель планов** есть; до «зрелого SaaS» не хватает **платёжного контура**, **достоверного usage metering**, **персистентных квот**, **webhooks**, **Enterprise SSO** и операционной оболочки (status, SLA-док).

## 4. Дорожная карта: приоритетные улучшения

Выбраны **три** приоритета, которые дают максимум эффекта при разумном объёме работ и совместимы с обеими моделями — **облачный SaaS** и **desktop-first с агентом**.

### P1 — Замкнуть контур «план ↔ использование ↔ блокировки»

- Реализовать **реальный `usage`** (счётчики задач за сутки, число профилей/кампаний, оценка хранилища) в БД или агрегатах по `tenant_id` / пользователю.
- На критичных эндпоинтах (создание задачи, добавление профиля, запуск кампании) — **проверка лимитов** и понятные ошибки `402/403/429` с телом для UI.
- Устранить расхождение лимитов **UI** (например, `tasks_per_day: 50` для Starter на [`BillingPage.tsx`](../frontend/src/pages/cabinet/BillingPage.tsx)) и **`_PLAN_LIMITS`** в БД (там `starter` = 30 в [`database.py`](../core/database.py)).

*Зачем:* без этого страница тарифов и биллинга **не соответствуют** поведению системы; конкуренты на SaaS-рынке здесь обычно сильнее **доверием к лимитам**.

### P2 — Платежи и смена плана (если цель — продавать подписку)

- Интеграция **Stripe или Paddle** (или региональный аналог): checkout, customer portal, webhooks `invoice.paid` / `subscription.updated`.
- Обновление поля `plan` и **audit** (частично уже есть события админа — можно расширить под биллинг).

*Зачем:* это главный разрыв с типичным SaaS с таблицы конкурентов.

### P3 — Webhooks и прозрачность пайплайна

- URL webhook на уровне tenant или проекта: события **`task.completed`**, **`task.failed`**, опционально **`render.progress`**.
- Стабильный **внешний id задачи** для идемпотентности клиентских интеграций.

*Зачем:* выравнивает NeoRender с ожиданиями от **API-first** продуктов (Shotstack, Creatomate, Renderly) и упрощает интеграции Zapier/Make без поллинга.

### Модель продукта (кратко)

| Модель | На что усилить в маркетинге и продукте |
|--------|----------------------------------------|
| **Cloud / SaaS** | Hosted очередь рендера (или гибрид), биллинг, webhooks, статус, документация API. |
| **Desktop / private agent** | Приватность, отсутствие очереди в чужом облаке, локальный NVENC, явная политика данных; SaaS-конкурентов брать как ориентир по **UX онбординга и ценам**, а не по архитектуре. |

Рекомендация: **не копировать** чисто облачный рендер, а **дополнить** текущий задел: сильная сторона NeoRender — **полный локальный контур + мультитенантный API**; слабая — **коммерческая оболочка** (оплата, usage, webhooks).
